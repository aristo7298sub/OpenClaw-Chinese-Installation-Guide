# 飞书会议自动化系统 — 完整部署指南

> **功能**：飞书视频会议结束后，自动抓取转录 → AI 生成会议纪要 → 创建任务并分配负责人 → 写入多维表格 → 推送通知到对应群聊（含 @mention）  
> **技术栈**：OpenClaw + lark-cli + Bash + systemd  
> **预计部署时间**：30–60 分钟（已有 OpenClaw 实例）/ 1.5–2 小时（从零开始）  
> **最后更新**：2026-04-03

## 1. 系统架构

```
┌─────────────────┐     会议结束      ┌───────────────────────┐
│   飞书视频会议    │ ──────────────▶  │  meeting-watcher.sh   │
└─────────────────┘                  │  (双模式检测)          │
                                     │  ├─ 事件订阅(WebSocket)│
                                     │  └─ 轮询兜底(60s)      │
                                     └──────────┬────────────┘
                                                │ 获取 transcript
                                                ▼
                                     ┌───────────────────────┐
                                     │  飞书录制 API          │
                                     │  recording → minutes  │
                                     │  → transcript.txt     │
                                     └──────────┬────────────┘
                                                │
                                                ▼
                                     ┌───────────────────────┐
                                     │  OpenClaw Agent        │
                                     │  (office-director)     │
                                     │  ├─ 生成会议纪要       │
                                     │  ├─ 提取待办任务       │
                                     │  ├─ 创建飞书任务       │
                                     │  ├─ 写入多维表格       │
                                     │  └─ 推送群聊通知       │
                                     └───────────────────────┘
```

### 1.1 数据流

1. **检测会议结束**：watcher 通过事件订阅 + 轮询双模式监听 `vc.meeting.all_meeting_ended_v1`
2. **获取转录**：`recording API` → 提取 `minute_token` → `vc +notes` 下载 `transcript.txt`
3. **触发 Agent**：将 transcript 文本注入 Agent prompt，Agent 按 `SOUL.md` 定义的工作流执行
4. **Agent 产出**：创建飞书任务 → 写入 Bitable → 发送 post 格式群消息（含 @mention + 任务链接）
5. **清理**：处理完成后自动删除 transcript artifact 目录

### 1.2 双模式检测机制

| 模式 | 实现方式 | 优点 | 缺点 |
|------|---------|------|------|
| 事件驱动 | `lark-cli event +subscribe --event-types vc.meeting.all_meeting_ended_v1` (WebSocket) | 实时、低延迟 | 依赖 WebSocket 连接稳定性，偶有漏事件 |
| 轮询兜底 | `lark-cli vc +search --start <10分钟前>` 每 60s 查一次 | 可靠、不漏 | 延迟最大 60s |

两者同时运行，通过 `meeting_id` 去重，确保每个会议只处理一次。

## 2. 前置要求

### 2.1 服务器

- Linux 服务器（Ubuntu 22.04/24.04 推荐），1 vCPU + 2GB RAM 即可
- 已安装 Node.js ≥ 22
- 已安装 `jq`（`sudo apt install -y jq`）
- 可选：`inotifywait`（`sudo apt install -y inotify-tools`），用于事件目录的高效监控

### 2.2 软件

| 组件 | 用途 | 安装方式 |
|------|------|---------|
| **OpenClaw** | AI Agent 运行时 | `sudo npm install -g openclaw@latest` |
| **lark-cli** | 飞书 API 命令行工具 | OpenClaw 自带，安装后位于 `~/.local/bin/lark-cli` |
| **jq** | JSON 解析 | `sudo apt install -y jq` |

### 2.3 飞书资源

- 飞书企业管理员权限（用于创建应用、审批权限）
- 一个飞书多维表格（Bitable）用于存储任务看板
- 至少一个群聊，且 Bot 已加入该群

## 3. 飞书应用创建与权限配置

> **关键概念**：建议为会议自动化创建**独立的飞书应用**，不要与主 Agent 共享应用。这样可以：
> - 隔离权限范围
> - 避免主 Agent 被长时间会议处理阻塞
> - 独立管理 OAuth 授权

### 3.1 创建飞书应用

1. 打开 [飞书开放平台](https://open.feishu.cn/app)
2. 点击 **创建自建应用**
3. 填写应用名称（如 "Office Director"）、描述
4. 记录 **App ID** 和 **App Secret**（后续配置 lark-cli 需要）

### 3.2 启用机器人能力

1. 在应用管理页 → **添加应用能力** → 勾选 **机器人**
2. 保存

### 3.3 配置权限

这是最关键的步骤。以下权限缺一不可：

#### 3.3.1 应用权限（Bot 身份）

在 **权限管理** → **API 权限** 中申请以下权限：

| 权限标识 | 说明 | 用途 |
|---------|------|------|
| `vc:meeting` | 获取会议列表 | 轮询已结束的会议 |
| `vc:meeting:readonly` | 读取会议详情 | 获取会议时长、参与者 |
| `vc:record:readonly` | 读取会议录制 | 获取录制 URL 和 minute_token |
| `im:message:send_as_bot` | 以 Bot 身份发消息 | 推送会议纪要到群聊 |
| `im:chat:readonly` | 读取群聊列表 | 获取 Bot 加入的群聊 |
| `im:chat.member:read` | 读取群成员 | 获取群成员 open_id |
| `contact:user.base:readonly` | 读取用户基本信息 | 搜索用户 |
| `task:task:write` | 创建/编辑任务 | 创建飞书任务并分配 |
| `bitable:app` | 读写多维表格 | 写入任务到 Bitable |

#### 3.3.2 用户权限（User 身份 — 用于转录导出）

> **坑点**：飞书妙记的转录导出 API 只支持 **用户身份调用**，Bot 身份无法获取 transcript。必须添加以下用户级权限：

| 权限标识 | 说明 | 用途 |
|---------|------|------|
| `minutes:minutes.artifacts:read` | 读取妙记产物 | 获取 transcript 文件 |
| `minutes:minutes.transcript:export` | 导出妙记转录 | 下载 transcript.txt |

#### 3.3.3 事件订阅权限（用于企业全量会议监听）

| 权限标识 | 说明 | 用途 |
|---------|------|------|
| `vc:meeting.all_meeting_ended` | 全企业会议结束事件 | 实时接收所有会议结束通知 |

### 3.4 发布应用

1. 权限配置完成后 → **版本管理与发布** → **创建版本** → **提交审核**
2. 企业管理员在 **飞书管理后台** 审批通过
3. 应用状态变为 **已启用**

### 3.5 将 Bot 加入群聊

在需要接收通知的每个飞书群聊中：
1. 打开群设置 → **群机器人** → **添加机器人**
2. 搜索并添加刚创建的应用 Bot

### 3.6 用户授权（User Token）

> **重要**：因为 transcript 导出需要用户身份，需要完成 OAuth 用户授权。

```bash
lark-cli auth login --app-id <APP_ID> --app-secret <APP_SECRET>
```

按提示在浏览器中完成 OAuth 授权流程。授权后 lark-cli 会缓存 user_access_token，后续自动刷新。

### 3.7 权限 JSON（批量导入用）

如果飞书管理后台支持权限 JSON 导入，可使用以下配置：

```json
{
  "scopes": [
    "vc:meeting",
    "vc:meeting:readonly",
    "vc:record:readonly",
    "vc:meeting.all_meeting_ended",
    "im:message:send_as_bot",
    "im:chat:readonly",
    "im:chat.member:read",
    "contact:user.base:readonly",
    "task:task:write",
    "bitable:app",
    "minutes:minutes.artifacts:read",
    "minutes:minutes.transcript:export"
  ]
}
```

## 4. OpenClaw 安装与 Agent 配置

### 4.1 安装 OpenClaw（如果尚未安装）

参考 [OpenClaw 完整安装指南](./OpenClaw-installation-guide.md)。核心步骤：

```bash
# 安装 OpenClaw
sudo npm install -g openclaw@latest

# 初始化（会创建 ~/.openclaw/ 目录和 systemd 服务）
openclaw onboard --install-daemon

# 确认 Gateway 运行
openclaw gateway status
```

### 4.2 配置飞书 Channel

在 OpenClaw 配置中注册飞书应用：

```bash
# 通过 OpenClaw 管理界面添加飞书 Channel
openclaw channels add feishu \
  --app-id <你的APP_ID> \
  --app-secret <你的APP_SECRET> \
  --name "office-director"
```

也可以直接编辑 `~/.openclaw/openclaw.json`，在 `channels` 部分添加飞书账户配置。

### 4.3 创建 Agent

```bash
# 通过 OpenClaw 管理界面创建 Agent
openclaw agents create \
  --id "office-director" \
  --name "Office Director" \
  --channel "feishu" \
  --account "office-director"
```

创建完成后，Agent 的 Workspace 目录位于：
```
~/.openclaw/workspace-office-director/
```

### 4.4 配置 lark-cli 认证

```bash
# 设置 lark-cli 使用新应用的凭据
lark-cli config set app-id <你的APP_ID>
lark-cli config set app-secret <你的APP_SECRET>

# 完成用户授权（transcript 导出必需）
lark-cli auth login
```

## 5. Agent Workspace 文件

Agent 的行为由 Workspace 中的文件定义。以下是完整的文件结构：

```
~/.openclaw/workspace-office-director/
├── SOUL.md                    # Agent 的核心行为定义
└── knowledge/
    ├── people.md              # 人员 open_id 映射表
    └── groups.md              # 群聊 chat_id 映射表
```

### 5.1 SOUL.md

> 这是 Agent 的"灵魂"，定义了它的职责、工具使用、消息格式等全部行为规范。

<details>
<summary>点击展开完整 SOUL.md</summary>

```markdown
# SOUL.md — Office Director

你是 {公司名} 的 Office Director（办公室主任），一个全自动化的企业运营 AI Agent。

## 核心职责

### 1. 会议纪要工作流（自动触发）
当 meeting-watcher 检测到新结束的会议并提供 transcript 时：
1. 阅读会议转录，生成简洁会议总结（要点、决策、讨论结果）
2. 提取所有待办任务（含负责人、截止日期）
3. 为每个任务执行：
   - `lark-cli task +create` 创建飞书任务
   - `lark-cli task +assign` 分配给负责人
   - `lark-cli base +record-upsert` 写入多维表格
4. 用 `lark-cli im +messages-send --as bot` 向**相关群聊**发送 post 格式通知
5. 通知中包含 @负责人（真实 @mention）、任务链接、多维表格链接

### 2. 知识库维护（自主执行）
每次被唤醒时，检查并更新 knowledge/ 目录下的知识文件：
- **群聊列表**: `lark-cli im chats list --as bot --page-all` → 更新 knowledge/groups.md
- **人员列表**: 遍历每个群的成员 `lark-cli api GET /open-apis/im/v1/chats/{chat_id}/members --as bot --page-all` → 更新 knowledge/people.md
- 如果发现新群或新人员，立即追加到知识文件中

### 3. 智能路由
根据会议参与者、会议主题，判断通知应发送到哪个群聊：
- 优先发送到与会议参与者相关的群
- 如果无法判断，发送到默认群
- 永远不要发送到不相关的群

## 工具使用规范

### lark-cli 关键命令
# 搜索用户
lark-cli contact +search-user --query "姓名"

# 列出 bot 加入的所有群
lark-cli im chats list --as bot --page-all --format json

# 发送 post 格式消息（支持真实 @mention）
lark-cli im +messages-send --as bot --chat-id "oc_xxx" --post '...'

# 创建任务
lark-cli task +create --summary "任务标题"

# 分配任务
lark-cli task +assign --task-id "xxx" --member-id "ou_xxx"

# 写入多维表格
lark-cli base +record-upsert --base-token "xxx" --table-id "xxx" @data.json

### 多维表格资源
- base_token: <你的base_token>
- table_id: <你的table_id>

## 行为准则
- **只发最终结果**：群聊只收到一条精美的 post 格式会议纪要
- **用中文沟通**：所有飞书消息用中文
- **精确高效**：不问不必要的问题，直接执行
- **出错处理**：某步骤失败继续执行其他步骤，最后汇报失败项
- **open_id 是应用级的**：不同飞书应用下 open_id 不同，必须查 knowledge/people.md

## 会议纪要消息模板（参考格式）

标题: 会议纪要 | {会议主题}

📅 时间：{日期时间} CST | 时长 {时长}
👥 参会人：{参会人列表}

📝 会议要点
{内容}

✅ 待办任务
• {任务描述}
  负责人：
  @{负责人1} @{负责人2}
  截止：{截止日期时间}

📌 查看任务  |  📊 任务看板

## 知识文件格式

### knowledge/people.md
| 姓名 | open_id | 部门 |
|------|---------|------|

### knowledge/groups.md
| 群名 | chat_id | 备注 |
|------|---------|------|
```

### 5.2 knowledge/people.md

Agent 通过此文件查找 @mention 所需的 `open_id`。首次需要手动初始化，之后 Agent 会自动维护。

```markdown
| 姓名 | open_id | 备注 |
|------|---------|------|
| 张三 | ou_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx | 部门A |
| 李四 | ou_yyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyy | 部门B |

> 最后更新: YYYY-MM-DD
```

**获取方法**：
```bash
# 列出 Bot 加入的所有群
lark-cli im chats list --as bot --page-all --format json | jq '.data.items[].chat_id'

# 获取某个群的成员列表
lark-cli api GET /open-apis/im/v1/chats/<chat_id>/members --as bot --page-all
```

> **注意**：`open_id` 是应用级别的，每个飞书应用看到的 `open_id` 不同。必须用当前 Bot 的身份去查询。

### 5.3 knowledge/groups.md

Agent 通过此文件选择通知目标群聊。

```markdown
| 群名 | chat_id | 备注 |
|------|---------|------|
| 默认群 | oc_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx | 默认群 |
| 项目A群 | oc_yyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyy | 项目 |

> 最后更新: YYYY-MM-DD
```

**获取方法**：
```bash
lark-cli im chats list --as bot --page-all --format json \
  | jq -r '.data.items[] | "| \(.name) | \(.chat_id) | |"'
```

## 6. Watcher 脚本部署

### 6.1 安装脚本

```bash
mkdir -p ~/.openclaw/scripts

# 将 meeting-watcher.sh 复制到服务器
# 方式一：从本地 scp
scp meeting-watcher.sh <user>@<host>:~/.openclaw/scripts/

# 方式二：直接在服务器上创建
nano ~/.openclaw/scripts/meeting-watcher.sh

# 设置可执行权限
chmod +x ~/.openclaw/scripts/meeting-watcher.sh
```

### 6.2 脚本配置项

所有配置通过环境变量覆盖，可在 systemd service 文件中设置：

| 环境变量 | 默认值 | 说明 |
|---------|-------|------|
| `MEETING_WATCHER_CHAT_ID` | (你的默认群 chat_id) | 默认通知群 chat_id |
| `MEETING_WATCHER_AGENT_ID` | `office-director` | OpenClaw Agent ID |
| `MEETING_WATCHER_REPLY_ACCOUNT` | `office-director` | 飞书账户名称 |
| `MEETING_WATCHER_MINUTES_WAIT` | `300` | 等待 transcript 生成的最大秒数 |
| `MEETING_WATCHER_POLL_INTERVAL` | `30` | transcript 轮询间隔（秒） |
| `MEETING_WATCHER_MIN_DURATION` | `10` | 忽略短于此时长的会议（秒） |
| `MEETING_WATCHER_RECONNECT_DELAY` | `5` | WebSocket 断线重连延迟（秒） |
| `MEETING_WATCHER_SCAN_INTERVAL` | `60` | 轮询兜底间隔（秒） |
| `MEETING_WATCHER_SCAN_WINDOW` | `600` | 轮询回溯窗口（秒） |
| `MEETING_WATCHER_LOG_TZ` | `Asia/Shanghai` | 日志时区 |

### 6.3 脚本关键实现细节

#### Transcript 获取流程

```
meeting_id
  → GET /open-apis/vc/v1/meetings/{meeting_id}/recording (bot 身份)
  → 从 recording.url 提取 minute_token
  → lark-cli vc +notes --minute-tokens {token} --format json (user 身份)
  → 验证 transcript 文件非空（飞书处理中文件可能为空）
  → 如果为空，删除重试，最多等待 300s
```

#### 身份切换说明

| API 调用 | 使用身份 | 原因 |
|---------|---------|------|
| 会议搜索 (`vc +search`) | bot | 需要企业级权限查全量会议 |
| 录制获取 (`recording API`) | bot | 需要 `vc:record:readonly` 租户权限 |
| 转录导出 (`vc +notes`) | user | 飞书妙记 API 仅支持用户身份 |
| 发送消息 (`im +messages-send`) | bot | 以 Bot 身份发送群消息 |
| 创建任务 (`task +create`) | bot/user | 均可 |

#### 去重机制

- 已处理的 `meeting_id` 追加到 `~/.openclaw/meeting-watcher-processed.txt`
- 文件自动截断，仅保留最近 500 条
- 事件驱动和轮询共享同一个去重文件

#### Artifact 清理

- 每次会议处理完成后，自动删除下载的 `artifact-*` 目录
- 如果 transcript 路径不在 `artifact-*` 目录下，仅删除 transcript 文件本身

### 6.4 完整脚本

完整脚本见 [`scripts/meeting-watcher.sh`](./scripts/meeting-watcher.sh)。

## 7. systemd 服务化

### 7.1 创建 Service 文件

```bash
mkdir -p ~/.config/systemd/user

cat > ~/.config/systemd/user/meeting-watcher.service << 'EOF'
[Unit]
Description=Meeting Watcher — 飞书会议结束自动触发纪要工作流
After=network-online.target openclaw-gateway.service
Wants=network-online.target
StartLimitIntervalSec=300
StartLimitBurst=5

[Service]
Type=simple
ExecStart=%h/.openclaw/scripts/meeting-watcher.sh
Restart=on-failure
RestartSec=10
Environment=PATH=%h/.local/bin:/usr/local/bin:/usr/bin:/bin
Environment=HOME=%h

# 日志
StandardOutput=journal
StandardError=journal
SyslogIdentifier=meeting-watcher

[Install]
WantedBy=default.target
EOF
```

### 7.2 启用并启动

```bash
# 重新加载 systemd 配置
systemctl --user daemon-reload

# 启用开机自启
systemctl --user enable meeting-watcher.service

# 启动服务
systemctl --user start meeting-watcher.service

# 查看状态
systemctl --user status meeting-watcher.service
```

### 7.3 确保 user service 在 SSH 断开后继续运行

```bash
# 启用 lingering（关键！否则 SSH 断开后 user service 会停止）
sudo loginctl enable-linger $(whoami)
```

### 7.4 日志查看

```bash
# 实时跟踪日志
tail -f ~/.openclaw/logs/meeting-watcher.log

# 通过 journalctl 查看
journalctl --user -u meeting-watcher -f

# 查看最近 50 行
tail -50 ~/.openclaw/logs/meeting-watcher.log
```

```bash
lark-cli vc +search --start "$(date -u -d '1 hour ago' '+%Y-%m-%dT%H:%M:%SZ')" --format json

# 5. 确认 Agent 能被调用
openclaw agent --agent office-director --message "你好，请回复 OK" --timeout 30

# 6. 确认 Bot 已加入目标群聊
lark-cli im chats list --as bot --page-all --format json | jq '.data.items[].name'
```

### 8.2 进行一次测试会议

1. 在飞书发起一个视频会议（需要开启**录制**）
2. 讲话超过 10 秒，提到至少一个待办任务
3. 结束会议
4. 观察日志：
   ```bash
   tail -f ~/.openclaw/logs/meeting-watcher.log
   ```
5. 预期在 1–3 分钟内看到：
   - `[轮询] 发现新会议: ...`
   - `获取录制 transcript...`
   - `录制就绪，minute_token: ...`
   - `Transcript 已下载且有内容: ...`
   - `触发 Agent (office-director)...`
   - `Agent 执行完成`
   - `已清理会议记录目录: ...`
6. 在目标飞书群聊中收到格式化的会议纪要消息

## 9. 运维与排障

### 9.1 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| Transcript 文件为空，一直重试 | 飞书妙记处理延迟 | 正常行为，脚本会重试最多 300s |
| `vc +notes` 返回 403 | 缺少 user-scope 的 minutes 权限 | 在飞书应用后台添加 `minutes:minutes.artifacts:read` 和 `minutes:minutes.transcript:export`，重新用户授权 |
| 录制 API 返回空 | 会议未开启录制 | 在飞书会议设置中开启自动录制 |
| Agent 执行超时 | LLM 响应慢 | 调大 `--timeout` 或检查模型 Provider |
| 消息中 @mention 无效 | 使用了错误的 open_id | open_id 是应用级的，需要用当前 Bot 身份查询 |
| 事件订阅无事件到达 | WebSocket 连接问题 | 正常降级到轮询兜底模式；检查 `vc:meeting.all_meeting_ended` 权限 |
| 服务重启后不运行 | 未启用 lingering | `sudo loginctl enable-linger $(whoami)` |

### 9.2 手动重试某个会议

如果某次会议处理失败，可以手动重新触发：

```bash
# 1. 从 processed 列表中移除该 meeting_id
sed -i '/<meeting_id>/d' ~/.openclaw/meeting-watcher-processed.txt

# 2. 重启 watcher（它会在下次轮询时重新处理）
systemctl --user restart meeting-watcher.service
```

### 9.3 重置 User Token

如果 user_access_token 过期：

```bash
lark-cli auth login
# 按提示在浏览器中完成 OAuth
```

### 9.4 日志轮转

watcher 日志会持续增长。建议配置 logrotate 或定期手动清理：

```bash
# 手动截断日志（保留最近 1000 行）
tail -1000 ~/.openclaw/logs/meeting-watcher.log > /tmp/mw.log && mv /tmp/mw.log ~/.openclaw/logs/meeting-watcher.log
```

## 10. 一键部署脚本评估

### 10.1 可自动化的部分

| 步骤 | 自动化可行性 | 说明 |
|------|------------|------|
| 安装 jq, inotify-tools | ✅ 完全可以 | `apt install -y` |
| 安装 OpenClaw | ✅ 完全可以 | `npm install -g openclaw@latest` |
| 创建目录结构 | ✅ 完全可以 | `mkdir -p` |
| 部署 watcher 脚本 | ✅ 完全可以 | 脚本内嵌或 curl 下载 |
| 部署 systemd service | ✅ 完全可以 | 模板生成 |
| 生成 SOUL.md 模板 | ✅ 完全可以 | 替换占位符 |
| 生成 knowledge 模板 | ✅ 完全可以 | 空表模板 |
| 启动服务 | ✅ 完全可以 | `systemctl --user enable --now` |

### 10.2 需要人工介入的部分

| 步骤 | 原因 |
|------|------|
| **创建飞书应用** | 需要在飞书开放平台 Web UI 操作 |
| **配置权限** | 需要在 Web UI 逐项勾选并提交审核 |
| **管理员审批** | 需要企业管理员在管理后台操作 |
| **将 Bot 加入群聊** | 需要群管理员在飞书 App 中操作 |
| **OAuth 用户授权** | 需要在浏览器中完成交互式授权 |
| **创建 Bitable** | 需要在飞书中手动创建多维表格 |
| **填写 App ID / Secret** | 安全信息需要用户手动输入 |

### 10.3 结论与建议

**可以做一键部署脚本，但它更适合叫"配置引导脚本"（Setup Wizard）**。

建议方案：一个**交互式安装脚本** `setup-meeting-automation.sh`，流程如下：

```
┌─ 前置检查 ─────────────────────────────┐
│  检查 Node.js / jq / OpenClaw 是否安装   │
│  缺少则自动安装                          │
└──────────────┬─────────────────────────┘
               ▼
┌─ 收集信息（交互） ─────────────────────┐
│  1. 飞书 App ID / App Secret           │
│  2. 默认群 chat_id                     │
│  3. Bitable base_token / table_id      │
│  4. Agent ID（默认: office-director）   │
│  5. 时区（默认: Asia/Shanghai）         │
└──────────────┬─────────────────────────┘
               ▼
┌─ 自动部署 ────────────────────────────┐
│  1. 创建 OpenClaw Agent               │
│  2. 配置 lark-cli 认证                │
│  3. 生成 SOUL.md（填充用户输入）        │
│  4. 初始化 knowledge/ 文件             │
│  5. 部署 meeting-watcher.sh            │
│  6. 创建 systemd service              │
│  7. 启用 lingering                    │
│  8. 启动服务                          │
└──────────────┬─────────────────────────┘
               ▼
┌─ 引导完成人工步骤 ────────────────────┐
│  打印清单：                           │
│  □ 确认飞书应用已创建并启用            │
│  □ 确认权限已审批                     │
│  □ 确认 Bot 已加入目标群聊             │
│  □ 运行 lark-cli auth login 完成授权   │
│  □ 发起测试会议验证                   │
└───────────────────────────────────────┘
```

**投入产出比**：如果只是团队内部 3-5 人使用，这份文档足够。如果要对外推广或服务多家客户，配置引导脚本值得投入。建议先用文档跑通几个客户安装流程，收集反馈后再做脚本。

### C. 安全注意事项

- **不要将 App Secret 提交到 Git 仓库**
- lark-cli 的 token 缓存文件默认在 `~/.config/lark-cli/`，权限应为 `700`
- systemd service 使用 user 级别服务，不需要 root 权限
- watcher 日志可能包含会议内容摘要，注意日志访问权限
